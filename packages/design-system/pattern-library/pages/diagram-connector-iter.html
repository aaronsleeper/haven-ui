Chevron-size variants — pick the proportion

Iter #4 shrinks the chevron because iter #3's 13×28 felt disproportionately large relative to the dot and line weight. These four variants vary chevron-width as a fraction of corner-radius (24 user-units) — pick the proportion that feels harmonious with the rest of the connector. All maintain FigJam's ~94° opening; only outer dimensions change.

A · 6×13 (radius/4)

Most delicate. Chevron is a subtle terminator, not a destination.

B · 8×17 (radius/3) ★

Recommended. Visible terminator; balanced with dot and line.

C · 10×21 (radius/2.4)

Mid-large. Reads as a clear arrowhead.

D · 13×28 (FigJam)

FigJam's outer dimensions (iter #3). Aaron felt this disproportionately large.

Note on the FigJam-to-corner-radius ratio question: FigJam's chevron is 13 wide vs corner-radius 24, ≈ ratio 1:1.85 (variant D). My read is that this ratio is FigJam's default arrowhead-to-arc relationship rather than a deliberate brand-aligned choice. Variants A–C explore tighter ratios that feel more harmonious with the connector's other geometry.

Iteration notes

What this page demonstrates: the mechanical-instantiation workflow. Designer authors a primitive once (FigJam, in this case Aaron's existing diagram); spec captures path data verbatim; haven-ui instantiates as stroked SVG with markers. The two renderings should be visually equivalent. Discrepancies are bugs in translation, not creative-judgment failures.

Iteration history:

  • Iter #1 (filled chevron-with-notch, 13×28, M 12 14 L 0 0 L 4 14 L 0 28 Z): connector body geometry validated by Aaron 2026-05-09; marker read ~10× heavier than FigJam intent because filled shapes don't match FigJam's stroked-line aesthetic.
  • Iter #2 (open stroked chevron, 13×14, M 0 0 L 13 7 L 0 14): right approach (drawn-by-virtue-of-being-the-same-line) but compressed — chevron read as too-narrowly-angled and the round caps at flank ends were too small to register at this opening width. ~57° opening vs FigJam's ~94°.
  • Iter #3 (open stroked chevron, 13×28, M 0 0 L 13 14 L 0 28): widened to FigJam's outer dimensions. Aaron's feedback: chevron disproportionately large; dot proportionally small; sharp ends (round caps too small relative to flank length); body's round cap protruded through chevron tip (because refX=13 put tip at body endpoint).
  • Iter #4 (current — open stroked chevron, 8×17, M 0 0 L 8 8.5 L 0 17): chevron sized at corner-radius/3 for proportional harmony; refX=0 so chevron extends past body endpoint instead of overlapping (tip covers line's end); dot bumped to r=3 (1.5× stroke-width). Variants row below shows 4 sizes for picking.

Known follow-ons (eyeball-confirm-pending):

  • Marker dimensions are 13×14 in user-space units. May still be too large for haven-ui diagrams at typical scale (560×260 viewBox vs FigJam's 1652×703). Consider a proportional scale-down option once base shape is brand-team-validated.
  • Start-dot is rendered as a 2-radius circle here. FigJam's version uses a slightly different shape (4-Bézier closed path); difference is invisible at this scale.

Next iteration steps (in order):

  1. Aaron eyeballs the side-by-side. If the haven-ui instantiation reads as visually equivalent (or better), the connector primitive locks at this geometry.
  2. If not equivalent: identify the specific discrepancy (likely fillet curves on marker corners), revise marker path, re-render, re-compare. Each round is measurable — visual diff or it's not the issue.
  3. Once locked: capture the final connector spec (path-data templates, marker path, color and stroke tokens) into data-visualization.md §2.4 as canonical primitive data. This becomes the reference the AI mechanically instantiates against.
  4. Move to next primitive (node? marker variants? substrate?). Same workflow: isolated iteration page, designer-authored exemplar, spec capture, lock.
