Diagram Text-in-Container — Iteration Ground

Isolated iteration page for the text-in-container primitive. The substrate-variant diagram-box bug (Aaron 2026-05-10, fixed at commit 7f2a086) revealed the deeper challenge: SVG has no auto-layout, so every text element needs manual x/y/text-anchor set correctly. Multi-line text, padding, container-width-aware alignment all live in the author's head. This page compares two approaches — pure-SVG with manual positioning vs. <foreignObject> HTML escape — across a substrate-style scenario and a multi-line stress test.

DECISION (2026-05-10): Approach B (foreignObject) is the canonical convention.

Diagrams need reliable text containment AND wrap. Approach A's silent-overflow failure mode (visible in the stress test below) disqualifies it as a default — unknown-future content length means real diagrams will break in production. Approach A is retained on this page for comparison + the why-not record. Trade-off accepted: foreignObject's static-asset-export quirks (rsvg / librsvg may mis-render). Acceptable as long as Cena diagrams render in modern browsers (haven-ui apps, vault HTML, slide-deck-via-browser-screenshot). Revisit and consider Option 2 (hybrid by content type) if export issues bite.

Scenario: substrate-style box (overline · heading · sublabel · pill)

Both approaches render the same substrate scenario at the same container size. Container is 400px wide; SVG viewBox uses 760×160 user-units (matching the diagram-box component's canonical size). The text content fits comfortably here.

A · Pure SVG (manual positioning)

Each text element gets explicit x, y, and text-anchor. The pill rect's center (640, 76) is computed by hand; the label's x="640" + text-anchor="middle" centers it. No reflow — content must fit the author's chosen positions.

B · foreignObject (HTML escape)

Single <foreignObject> wraps the box's interior. Layout via flexbox + padding. Pill is just a styled <div>. No manual x/y math; no text-anchor. Reflow is automatic if content changes.

Stress test: narrow container forces wrap decision

Same heading text, narrow rect (240 user-units wide instead of 680). Pure SVG runs past the rect's right edge silently. foreignObject wraps via CSS line-break. This is where the two approaches genuinely diverge.

A · Pure SVG (no wrap)

Red dashed line marks the rect's right edge. Heading and sublabel BOTH cross it freely — the SVG viewBox doesn't clip; text just keeps going. The SVG has a subtle warm-tint background so overflowing white-on-dark text remains visible past the rect edge (without it, white text on white wrapper would appear to clip at the rect — a misleading visual). Author must shorten content, switch to <tspan> with hand-computed dy offsets, OR use approach B.

B · foreignObject (auto-wrap)

Same content; CSS line-wrap fires inside the foreignObject. Box height bumped from 120 to 160 user-units to give wrap room — if content exceeds even that, add overflow:hidden or grow the rect.

Responsive test: same SVG at 3 container widths

The substrate scenario rendered at 240/400/640px container widths. Pure SVG (top row) and foreignObject (bottom row). Both should hold their internal layout regardless of rendered size — the SVG viewBox handles scaling. foreignObject's HTML scales too, but font sizes inside are in CSS px which scales with the SVG viewBox transform.

Trade-offs at a glance

Concern A · Pure SVG B · foreignObject
Single-line text positioning Manual x/y/text-anchor per element. Foot-gun if CSS overrides attributes (the substrate bug). CSS handles it (text-align, padding). No SVG-specific authoring rules.
Multi-line text wrap Manual via <tspan dy>. Wrap point hand-computed; doesn't reflow. Automatic via CSS line-wrap. Reflows to container width.
Padding Hand-computed offsets from rect edges. CSS padding property. Standard.
Layout primitives (flex, grid) None. Compute every position by hand. Full HTML/CSS layout system available.
Static-asset export (PNG, PDF, image embed) Reliable across all SVG renderers and image converters. Quirky. Many SVG-to-PNG tools (rsvg, librsvg, headless browsers, image embedders) drop or mis-render foreignObject content.
Browser rendering Universal — every browser and every SVG implementation. All modern browsers (Chrome, Firefox, Safari, Edge). Some niche tools may skip.
Accessibility Standard SVG <text> — well-supported by screen readers; aria-label on root. HTML inside is screen-reader accessible, but Safari has historic bugs reading foreignObject content.
CSS bleed risk SVG presentation attributes mostly isolated; CSS can still override (the substrate bug). HTML inside foreignObject inherits surrounding CSS — global Tailwind reset, body styles, etc. all apply unless reset.
Conform-gate friendliness Class-based components.css rules apply. Existing gates (token, css-family) work. Inline styles inside foreignObject bypass class system. Hard to gate.
Authoring effort Higher. Every text decision is a math problem. Lower. Same as authoring HTML.

Decision (Aaron 2026-05-10): Approach B is canonical. Reliable containment + wrap is a hard requirement for diagram authoring — content length is not known in advance, and silent overflow with no wrap means broken diagrams in production. A's foot-gun is structural, not fixable by author discipline.

The trade-off accepted: foreignObject's static-asset export is quirky (rsvg / librsvg / image-embedders may drop or mis-render). Acceptable because Cena's diagrams are consumed by modern browsers (haven-ui apps, vault HTML, slide decks via browser screenshot). If export-to-raster reliability becomes load-bearing (e.g., for a client-facing PDF pipeline), revisit and consider the hybrid path: A for fixed-size single-line labels, B for any text that might wrap.

The substrate-bug fix shipped in 7f2a086 (text-anchor in markup not CSS) still stands for any legacy SVG-text usage that hasn't migrated to foreignObject yet — most importantly, the canonical diagram-box.html substrate variant. Migration to foreignObject is a separate task; this iteration locks the convention for new authoring.
