Administration — PHQ9 at baseline and repeat screenings

DRAFT • v0.1 • pending Marrero + Healthcare Data Governance review

Cena's PHQ9 administration flow drives a CITI-trained Care Coordinator through baseline and repeat-screening administration milestones, captures the score on the canonical participant record, applies a protocol threshold check, and routes positive results same-day to the escalation use case (no deferral) while sub-threshold results route to longitudinal monitoring. The UConn PI provides scope-of-practice oversight as a partner-as-watcher annotation on the administration step.

Elements

  • Trigger — screening milestone reached — system enquiry step in the SYSTEM lane; the Cena scheduling system tracks each enrolled patient's IRB-protocol screening schedule (baseline + repeat-screening) and notifies the assigned care coordinator.
  • Action — CC administers PHQ9 — human action step in the HUMAN lane; the CITI-trained Care Coordinator administers the screening to the patient on the Cena assessment-forms platform; channel is in-person at baseline and remote (pending Marrero) at repeat-screenings; CT scope-of-practice is pending Healthcare Data Governance confirmation; the UConn PI watches scope-of-practice from the PARTNER lane.
  • Enquiry — score captured to participant record — system enquiry step in the SYSTEM lane; the Cena assessment-forms platform records total score, per-item responses, and administration metadata to the participant's assessment-results subcollection; this is the canonical record-of-score for all downstream reasoning.
  • Decision — threshold check — agent decision step in the AGENT lane; the agent compares the captured score against the protocol-defined positive threshold (pending HDG, same gap shape as escalation use case fragment 01) and the Q9 (suicidal ideation) override; classification is policy-derived, not clinical-judgment per-case.
  • Hand-off — positive routes to escalation; sub-threshold routes to monitoring — agent hand-off step in the AGENT lane; positive results route same-day with no deferral to the escalation-phq9-positive use case (cross-use-case routing); sub-threshold results route passively to longitudinal monitoring; remote-channel first-action specifics are a structural gap per catalog open question #4.

Relationships

  • Trigger → CC administers: system-to-human cross-lane flow (default arrow).
  • CC administers → score captured: human-to-system cross-lane flow (default arrow); the questionnaire is complete and the platform writes the canonical record.
  • Score captured → threshold check: system-to-agent cross-lane flow (default arrow).
  • Threshold check → hand-off: within-AGENT-lane flow (default arrow).
  • Hand-off → escalation use case (positive): cross-use-case routing (emphasized arrow); the case leaves the administration frame and enters the escalation-phq9-positive frame at its trigger fragment.
  • Hand-off → longitudinal monitoring (sub-threshold): passive cross-use-case routing (muted dashed arrow); the canonical record is left on the participant record for next-protocol-milestone review.
  • CC administers — UConn PI watches: cross-lane watcher annotation rendered as a dashed line from the CC's box down into the PARTNER lane; the PI does not produce a step but provides scope-of-practice oversight on the CC's administration.

Three uncertainty annotations are surfaced visibly on the diagram: CT scope-of-practice + UI build state on fragment 02 (TBD pending HDG), threshold value on fragment 04 (TBD pending HDG — deliberately the same modifier as escalation fragment 01 for consistency), and remote-channel first-action specifics on fragment 05 (GAP per catalog open question #4). The downstream escalation use case carries its own gaps (BHN role, clinical-lead identity, severity tier boundaries) which are inherited via the cross-use-case routing edge but not duplicated in this diagram.
