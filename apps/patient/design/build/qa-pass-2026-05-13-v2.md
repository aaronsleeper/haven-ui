## Cross-cutting findings (affect multiple screens)

### CRITICAL — `.message-new-pill` is `position: fixed` regardless of consumer

**File:** `packages/design-system/src/styles/tokens/components.css:6143–6151`

```css
.message-new-pill {
  @apply fixed left-1/2 -translate-x-1/2 z-30;
  @apply bg-primary-500 text-white rounded-full text-sm px-3 flex items-center gap-1.5 shadow-md;
  bottom: 136px;
  height: 32px;
}
```

The class is consumed in **two semantically different places**:

1. **Dashboard `Recent message` card** (`apps/patient/src/screens/dashboard/index.tsx:117`) — meant as an inline corner-anchored unread indicator on a preview card
2. **Messages thread bubble** (`apps/patient/src/screens/messages/index.tsx:177`) — meant as a "new beat" pill at the top of the latest unread care-team bubble

The CSS comment says: *"Floating 'new message' pill — sits above compose bar when new messages arrive. Bottom offset: 64px nav + 64px compose = 128px"* — so the rule was authored for the **Messages screen new-message floater pattern** (different use case from the rendered ones). All three consumers — dashboard preview, in-bubble, and the implied floater — share one class with `position: fixed`. Result: every "New" pill on every screen floats at `bottom: 136px, left: 50%` in viewport coordinates, decoupled from its DOM position.

**Visible consequences (verified by headless inspection):**
- Dashboard (`/`): "New" pill renders at viewport `left: 640, bottom: 552` — directly on top of the Delivery card, not on the message preview card. Looks like a misplaced badge floating across the layout.
- Messages (`/messages`): same — pill floats at bottom-center over empty space below the thread.
- After reply persisted: pill correctly disappears via `suppressUnread` logic.

[Source: [WCAG 2.1 SC 1.3.1 Info and Relationships](https://www.w3.org/WAI/WCAG21/Understanding/info-and-relationships.html) — the visual "this pill is about that message" relationship is broken; sighted users cannot associate the pill with the message.]

**Fix:** Decompose into two classes:
- `.message-new-pill` (inline variant) — `inline-flex bg-primary-500 text-white rounded-full text-xs px-2 py-0.5` for the dashboard preview corner indicator
- `.message-thread-new-floater` (or extend `.message-new-pill--floating`) — the existing fixed-position rule, used only on Messages thread when scroll-restoration patterns are added

Until the floater pattern is actually implemented, the simplest fix is to drop `position: fixed` from the base class and add a modifier class for the (unimplemented) thread-bottom usage.

**Severity:** Critical (demo-blocker). The pill is the visual cue that Sarah's message just arrived — the demo opening beat. Currently it appears to float over unrelated UI.

### CRITICAL — Patient bubble (`.message-bubble-out`) fails WCAG 1.4.3 AA contrast

**File:** `packages/design-system/src/styles/tokens/components.css:6112–6116`

Computed values from the rendered DOM:
- Background: `rgb(58, 132, 120)` (`bg-primary-500`, hex `#3A8478`, teal-500)
- Foreground: `rgb(255, 255, 255)` (white)
- Font size: 14px, font weight: 400 (normal)
- **Contrast ratio: 4.42:1**
- WCAG 1.4.3 AA threshold for normal text: 4.5:1
- **Result: FAILS by 0.08**

This is the patient bubble — the demo headline element. Aaron's Q1 settled framing (patient teal, care team sand) was made before the contrast pass. The settlement is independently defensible on positioning grounds; the contrast failure is a separate axis.

[Source: [WCAG 2.1 SC 1.4.3 Contrast (Minimum)](https://www.w3.org/WAI/WCAG21/Understanding/contrast-minimum.html). "Large text" exception (3:1 floor) requires ≥18pt regular OR ≥14pt bold — 14px regular is 10.5pt regular, well below the large-text threshold.]

**Two paths to fix without compromising the brand frame:**

1. **Darken the teal one shade** — swap `.message-bubble-out` from `bg-primary-500` (`#3A8478`) to `bg-primary-600` (`#2C6A60`, used on the Send button which clears AA at 6.25:1). Brand-fidelity reading: the bubble being one shade darker than the Send button breaks the Send's "this is the commit moment teal" reservation by making the bubble compete with it less, not more. Trade-off: the brand-fidelity verdict noted the bubble-as-teal already competes with Send-as-teal; darkening the bubble strengthens that competition since the Send button stays teal-600. Better than the failing-AA path.
2. **Bold the bubble text** — change `.message-bubble-out p` to `font-weight: 700`. At 14px bold, the contrast threshold drops to 3:1 (large-text exception applies to bold ≥14pt). Trade-off: aesthetic shift, may read shouty.

**Recommended:** path 1 (darken to primary-600). Single CSS change, no typography shift, clears AA and approaches AAA-large-text (4.5:1 for bold 14pt). Re-verify after change.

**Care-team bubble (`.message-bubble-in`):** `rgb(230, 228, 224)` on `rgb(37, 33, 29)` = **12.59:1** — passes AA + AAA. No change needed.

**Other contrast pairs verified:**
- Message sender label (12px, sand-600 on sand-50): 4.68:1 — passes AA, fails AAA (7:1 for normal text)
- Message timestamp (12px, sand-600 on sand-50): 4.68:1 — passes AA, fails AAA
- Trend "Improving" badge: 6.21:1 — passes AA
- Trend "Stable" badge (sand-500): 4.88:1 — passes AA
- Send button (teal-600 on white, 11.11px Medium): 6.25:1 — passes AA

### MODERATE — Sand-400 and sand-500 muted text fail WCAG 1.4.3 AA at small sizes

Multiple screens use `text-sand-400` and `text-sand-500` for secondary/muted copy at 12–14px regular. Computed contrast against the sand-50 page background:

| Element | Color | Size | Contrast | AA result |
|---|---|---|---|---|
| `.text-sand-400` on sand-50 ("Updated 2 days ago", "Need help?" footers) | `#A8A39B` | 12px | **2.51:1** | **FAIL** |
| `.text-sand-500` on sand-50 ("Your progress, your story.", subtitles) | `#958E85` | 14px | **3.10:1** | **FAIL** |

Sand-400/500 are widely used as the muted-text register across Dashboard, Health, Care, Settings, Welcome, Consent, Preferences, GAD-7. Every page-subtitle (`text-sand-500 mt-1`) directly under a `.page-title` fails AA. Every "last updated" / "footer help" / "skipping is okay" line in `text-sand-400` fails AA harder.

[Source: [WCAG 2.1 SC 1.4.3 Contrast (Minimum)](https://www.w3.org/WAI/WCAG21/Understanding/contrast-minimum.html) — 4.5:1 floor for normal text under 18pt regular / 14pt bold.]

**System-level fix:** retune sand-400 to ~`#8A8278` (≈4.5:1) and sand-500 to ~`#6D665E` (≈6:1) at the token level. Or apply selectively: any `text-sand-400` consumer on a page-level background needs to be sand-600+ (`#777069` clears AA at 4.68:1, verified on the timestamp computation).

**Severity:** Moderate. Not a demo-blocker (Dieckhaus's team unlikely to run Axe live), but a defensibility blocker for any subsequent accessibility audit. Token-level retune is one-time work.

### MODERATE — Sidebar Messages badge does not reflect reply state

**Files:** `apps/patient/src/App.tsx:68` (`const unreadCount = 1;`), `apps/patient/src/components/Sidebar.tsx:42–49`

The sidebar Messages badge shows `1` unread regardless of reply state. The dashboard "New" pill correctly clears after the patient replies (via `MESSAGE_REPLY_KEY` localStorage check), but the sidebar badge stays at `1`. State coherence gap: the dashboard says "no new messages" while the sidebar says "1 unread."

For the demo, the sidebar badge will remain red `1` even after the patient has replied to Sarah. That contradicts the dashboard's cleared-pill state.

**Fix:** in `App.tsx`, replace the hardcoded `const unreadCount = 1` with a read against `MESSAGE_REPLY_KEY`:
```ts
const [hasReplied, setHasReplied] = useState(false);
useEffect(() => { /* read from localStorage */ }, []);
const unreadCount = hasReplied ? 0 : 1;
```
Or, the cleaner pattern: extract a `useUnreadCount()` hook that owns the logic, used by both AppShell and any future inbox consumer.

**Severity:** Moderate. Visible inconsistency during the demo opening beat (cold-start dashboard says "new" → patient replies → returns to dashboard → pill clears but sidebar badge remains). A Dieckhaus team member who notices will read it as "the agent doesn't know its own state."

### MODERATE — Focus management after Cancel/Send drops focus to `<body>`

**File:** `apps/patient/src/screens/messages/index.tsx:222–235`

When the patient opens the reply composer (focuses textarea), types, then clicks Cancel or Send, focus does not return to the compose trigger button. Headless verification:
- After Cancel: `document.activeElement.tagName === 'BODY'`
- After Send: `document.activeElement.tagName === 'BODY'`

The compose button is recreated when the panel collapses, so React's auto-restore-focus doesn't fire. Keyboard users lose their position and have to tab from the top.

[Source: [WCAG 2.1 SC 2.4.3 Focus Order](https://www.w3.org/WAI/WCAG21/Understanding/focus-order.html). The pattern "open panel → cancel → focus returns to trigger" is the WAI APG dialog/disclosure pattern; the absence of focus restoration is a usability failure for AT users.]

**Fix:** in `handleSend` and the Cancel onClick, set a flag that triggers an effect to focus the compose button after re-render:
```tsx
const composeRef = useRef<HTMLButtonElement>(null);
useEffect(() => {
  if (!replyExpanded && shouldRestoreFocus) {
    composeRef.current?.focus();
    setShouldRestoreFocus(false);
  }
}, [replyExpanded, shouldRestoreFocus]);
```

**Severity:** Moderate. Affects keyboard users only; sighted mouse users unaffected. Not demo-blocking but cheap to fix.

## Accessibility verdict on the kept bubble register

Aaron settled Q1 in favor of **patient-teal protagonist** before the contrast pass was run. The settlement is independently defensible on positioning grounds (patient-as-protagonist for agent-augmented care). The contrast cost is now measured:

| Bubble | Background | Foreground | Size | Weight | Ratio | AA |
|---|---|---|---|---|---|---|
| Patient (out) | `#3A8478` (primary-500) | white | 14px | 400 | **4.42:1** | **FAIL** |
| Care team (in) | `#E6E4E0` (sand-100) | `#25211D` (sand-900) | 14px | 400 | 12.59:1 | PASS |

The patient bubble fails WCAG 1.4.3 AA by 0.08. This is binary in compliance terms but cosmetic in visible impact — the text reads legibly to typical-vision users. The defensibility cost is real: any Axe / Wave / Lighthouse audit flags this, and Cena will face the question.

**Resolution paths (re-iterated):**
1. **Darken to `primary-600` (`#2C6A60`):** brings contrast to ~6.25:1 (verified via Send button parity), no brand frame disturbance, single CSS change. Minor side-effect: bubble visually competes less with the Send button it produces (which is already teal-600). Brand-fidelity F1 noted the bubble-and-Send-both-teal issue as residual cost of Q1; darkening the bubble does NOT resolve that — they remain the same shade. Acceptable.
2. **Keep `primary-500`, bold the bubble text:** 14px bold qualifies for large-text threshold (3:1) which `primary-500/white` clears at 4.42:1. Bolder bubble text changes the demo register visibly. Less preferred.
3. **Switch to a darker brand teal token (introduce `primary-550` or similar):** infrastructure overkill for one bubble.

**Recommendation:** path 1. Single change to `components.css:6113` swapping `bg-primary-500` to `bg-primary-600`. Re-verify contrast after change (target ≥4.5).
