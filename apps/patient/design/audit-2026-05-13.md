## Headline

Existing patient app is in much better shape than the "feels off" reaction suggested. Every semantic class used by the existing screens already exists in the current PL — `delivery-status-card`, `message-bubble-*`, `notif-item-*`, `trend-card`, `field-row`, `segmented-control`, `toggle`, `card`, `list-group`, `kv-table`, `modal-panel` are all confirmed in `COMPONENT-INDEX.md`. The "off" feeling traces to two specific drifts: (1) every screen's `<h1>` uses an inline typography literal (`text-[27.65px] font-serif font-medium text-sand-900`) instead of a semantic class, which makes the headers feel one-offs across screens; and (2) the shell is mobile-only by design — wireframe spec says "desktop renders centered at 430px max-width." Desktop responsiveness is a real, load-bearing decision (not a per-screen polish).

The biggest acceleration is here. We're not at zero. We're patching small drift on 5 already-built screens, building onboarding from scratch, and (optionally) building meal-ordering as the most visceral storytelling moment.

## Load-bearing decisions surfaced

### 1. Responsive shell — three options

The wireframe spec (`patient-mobile-shell.md` line 13) is explicit: "Mobile (320-430px); desktop renders centered at 430px max-width." `MobileShell.tsx` enforces this with `max-w-[430px] mx-auto`. The documented design intent is **patient app is mobile-only; desktop shows the same mobile UI centered at 430px against a sand-100 background**.

Your ask was "responsive version that looks good on desktop as well as mobile." Three ways to satisfy that:

- **A — Honor mobile-only design; improve desktop framing.** Keep 430px centered. Make the surrounding desktop ambient feel intentional (maybe a subtle phone-frame mockup, maybe a clean sand-100 with the patient's name/Cena logo). Low risk. Honors the documented design intent and the patient reality (phones).
- **B — Build true responsive shell.** Add a desktop breakpoint that expands. Use `layout-app-shell.html` (which exists in PL with sidebar — `app-sidebar`, `app-sidebar-nav`, etc.) as the desktop variant; current `MobileShell` becomes the narrow-viewport variant. Higher work; overrides documented design intent; the result is a "patient app that's also a desktop app."
- **C — Demo wrapper.** Keep mobile design as canonical code. For Dieckhaus demo, present the running mobile app in a phone-frame mockup on the desktop screen. Zero code change. The demo looks like a phone, because it IS a phone app.

My read: **A or C** depending on whether the demo screen should "be the product" (A) or "show the product" (C). B is the most work and creates ongoing maintenance for a second shell. Worth your call before any patching starts — the call affects every screen.

### 2. Header typography pattern

Every screen's `<h1>` is the same inline literal: `text-[27.65px] font-serif font-medium text-sand-900`. That's the wireframe's Heading/01 spec (Lora 27.65px Medium). Extract once to `components.css` as a semantic class, replace in 5 places, done. **Recommendation:** patch this in the cleanup pass. Adds the "consistent" feeling fast.

### 3. Sparkline implementation in `/health`

Trend cards in `My Health` have stubbed sparkline canvases (`TODO v1.1`). For the Dieckhaus demo, the trend story is compelling but needs visible movement on the cards. Three options:
- Wire Chart.js sparklines with demo data (faithful to spec; ~couple hours)
- Inline SVG with hardcoded paths (lower fidelity, faster)
- Leave the gray rectangles (lowest fidelity, demo feels half-done)

**Recommendation:** wire Chart.js with demo data. The trend chart is a centerpiece moment.
