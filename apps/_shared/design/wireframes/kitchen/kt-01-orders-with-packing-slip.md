# kt-01-orders-with-packing-slip: Orders + Packing Slip + Status Thread (Worked Example)

**Application:** Kitchen App
**Use Case(s):** KT-SHELL-02, KT-SHELL-03 (`apps/_shared/design/kitchen-shell-use-cases.md`)
**User Type:** Kitchen Staff (Carlos)
**Device:** Desktop primary (kitchen workstation); tablet (≥768px) supported
**Route:** `/orders/{orderId}`

This is the worked example for the kitchen slice (Gate 1 G1.2 minimum): orders queue + one open order with packing slip + status thread. Demonstrates the per-app minimum inside the universal agentic shell. Per Gate 2-prep decision 2, the right pane (status thread + status-progression buttons) auto-expands when an order opens even below 1240px.

---

## Page Purpose

Carlos clicks an order in the left pane. The center loads the packing slip — the canonical per-order detail view. The right pane shows the order's status thread + the status-progression buttons (Carlos's primary action surface). Carlos preps, packs, quality-checks, and dispatches with one tap each. Allergen flags are red, bold, and impossible to miss. The packing slip never shows clinical data — first name + last initial, delivery address, meal contents, allergen flags, delivery window only.

## Interaction Specifications

### Click an order in left pane
- **Trigger:** Click any `queue-item` (this state implies order is already selected; clicking another would change selection)
- **Feedback:** N/A in this state

### Tap "Start prep" (when in Pending state) [REVISED]
- **Trigger:** Tap `btn-primary btn-lg` "Start prep" in right pane
- **Feedback:** Button briefly inverts; success spinner ms-level; left pane regroups order from Pending → Prepping section; status badge updates in center record-header
- **Navigation:** Order stays active; right pane shows new status_change event "9:42 AM · [Carlos] started prep"
- **No undo** per Gate 2 decision 1. Tap-to-advance is the commit; reverting a status would create real-world inconsistency.
- **Error handling:** Status-change failure → `alert-error` inline below action row "We couldn't update the status. Tap to try again." + retry; order stays Pending

### Tap "Mark packed" (when in Prepping state)
- **Trigger:** Tap `btn-primary btn-lg` "Mark packed"
- **Feedback:** Same animation; left pane regroups Prepping → Packed; record header status update; thread logs event
- **Navigation:** Order stays active
- **Error handling:** Same as Start prep

### Tap "Quality checked" (when in Packed state, MANDATORY)
- **Trigger:** Tap `btn-primary btn-lg` "Quality checked"
- **Feedback:** Same animation; left pane regroups Packed → Quality Checked; record header status update; thread logs
- **Navigation:** Order stays active
- **Error handling:** Same; cannot skip to Dispatched without this step (UI enforces)

### Tap "Dispatch" (when in Quality Checked state) [REVISED]
- **Trigger:** Tap `btn-primary btn-lg` "Dispatch"
- **Feedback:** Confirm dialog: `overlay-confirm-dialog`
  - Title: "Dispatch order?"
  - Body: "Patient SMS will send automatically. There's no undo."
  - Buttons: "Cancel" (outline) + "Dispatch" (primary teal — kitchen commitment)
- **Navigation:** On confirm: status advances to Dispatched; SMS auto-sends to patient (templated by Ava in patient's language pref — "Your meal is on the way! Expect delivery 11:00-1:00." in EN/ES); thread logs event "Dispatched at HH:MM · Patient notified"; left pane regroups
- **No undo** on dispatch.
- **Error handling:** SMS-send failure does NOT roll back dispatch (separate channel); shows secondary toast "Patient SMS failed — please call patient at (555) 123-4567" with phone number revealed

### Tap "Flag issue"
- **Trigger:** Tap `btn-outline btn-sm` "Flag issue" below thread events
- **Feedback:** `overlay-modal` opens
  - Type select: `combobox` with "Ingredient shortfall" / "Recipe ambiguity" / "Allergen concern" / "Other"
  - Note textarea (required, min 10 chars)
  - Buttons: Cancel + Submit
- **Navigation:** On submit: issue posts to order thread + coordinator queue (urgent if Allergen); modal closes; toast confirmation
- **Error handling:** Validation on type + note; modal-level error.

### Tap an existing thread event
- **Trigger:** Click a `thread-msg-tool` (collapsible)
- **Feedback:** Tool detail expands inline
- **Navigation:** Stays in current view

### Resize / collapse right pane
- Per shell-kt-kitchen interaction spec

- Left: orders grouped; #4287 active in Prepping section

### Default (Order in Quality Checked)
- Status thread shows QC event; "Dispatch" button active (with confirm dialog)

### Default (Order in Dispatched)
- Status thread shows dispatch event + SMS sent confirmation; no further next-status (delivery is auto-advanced from dispatcher)
- Action row collapses to summary "Dispatched 11:08 AM · Awaiting delivery confirmation"

### Default (Order in Delivered)
- Status thread shows delivered event; action row hidden; only "Flag issue" remains
- Order moves to Delivered section in left pane

### Allergen-hold state
- If Allergen concern flagged at packing time, "Dispatch" button is disabled with tooltip "Hold until coordinator + RDN sign off"

- Left as shell-kt-kitchen
- Center: packing-slip skeleton — `skeleton` for header + 3 `skeleton-text` blocks for meal contents + `skeleton` block for allergen row
- Right: 3 `skeleton` rows for thread events; action row buttons in skeleton state

### Error State
- Center: `alert-error` if packing slip fails to load
- Right: `alert-error` if thread fails to load; status-progression buttons disabled until thread + order loaded

- Print affordance flagged for v1.1 — kitchen may want to print physical slips for line cooks; defer

## Accessibility Notes

- `<main aria-label="Order #4287 packing slip">` wraps center
- Allergen flags: large enough to scan from across the kitchen; text contrast WCAG AAA (red text on white background); icon + glyph + ALL CAPS label combine for redundant signal
- Status-progression buttons: 56px+ tall (gloved-hand-friendly); each has visible label + icon + `aria-label` describing the transition ("Mark packed — advance order from Prepping to Packed")
- Disabled buttons: visible (not hidden) with `aria-disabled="true"` + tooltip explaining why disabled
- Color-as-status: status badges use color + label + icon; allergen flags use color + uppercase label + glyph
- Decorative icons `aria-hidden="true"`
- Touch targets 44px+ everywhere; bottom-nav and primary action buttons 48-56px for tablet/gloved usage

## Bilingual Considerations

- Kitchen app EN-only at v1
- Patient first name + last initial render raw (no translation)
- Delivery address renders in original locale format (US addresses for US kitchens)
- Future Spanish adoption is a string update

## Open Questions

- Patient first-name + last-initial format: does kitchen need any other identifier (order # is primary)? Confirm "Maria R." is sufficient for picker accuracy without exposing PHI. Backup is the order number.
- Allergen flag format: "⚠ NUT-FREE" (with warning glyph + uppercase) vs. "⚠ Contains nuts" — the former is what to AVOID (the patient can't eat nuts), the latter is what's IN the dish. The packing-slip semantics is "patient cannot have" / restriction — recommend "NUT-FREE", "DAIRY-FREE" labels (the constraint, not the contents). Confirm.
- "Dispatch" button SMS auto-send: is the SMS body templated, or does Carlos compose? Recommend templated by Ava with patient-language pref ("Your meal is on the way! Expect delivery 11:00-1:00." in patient's language). Confirm.
- Quality-check button: should it require a checkbox / signature ("I verified portion + allergen"), or is the tap itself the verification? Recommend tap-as-verification at v1 (Carlos's name auto-attaches to the status_change event); add explicit checklist in v1.1 if quality issues arise.
- Print affordance for packing slip — needed at v1? Recommend deferred to v1.1; kitchens that need physical slips can browser-print for now.
- "Allergen hold" UX when coordinator + RDN sign off: how does the hold release? Coordinator approval → status-change event in thread → Dispatch button re-enables. Confirm flow.

## New Components Flagged

### `[NEW COMPONENT: packing-slip]` — Tier 1 primitive, brand-fidelity-weighted, 4-expert panel required

**Anatomy:**
- Container: `packing-slip` semantic class (sand-100 card-like envelope, `border-radius/md` 11px, `elevation/02`, padded `p-6`)
- Header section: `packing-slip-header` — patient first name + last initial (Lora display, Heading/02), delivery address (Body/02), delivery window (Body/02 Semibold)
- Allergen flag row: `packing-slip-allergens` — flex row of `packing-slip-allergen-flag` items
  - Each `packing-slip-allergen-flag`: large pill (32px+ tall), `bg-rose-16` + `text-rose-04` + `border-rose-14` + uppercase label + leading `fa-triangle-exclamation` glyph
  - Multi-restriction combos render as one combined pill (e.g., "GLUTEN-FREE + DAIRY-FREE") or stacked depending on count
- Meal contents list: `packing-slip-meals` — vertical list of `packing-slip-meal` items
  - Each `packing-slip-meal`: meal name (Body/02 Semibold) + portion / packing notes (Body/04 muted) + recipe ID (Body/04 muted, tabular)
- Delivery instructions section (optional): `packing-slip-instructions` — Body/03 muted block

**Why new:**
- No existing PL primitive composes the patient-id-restricted + allergen-prominent + meal-list pattern
- Allergen prominence is **safety-critical** — it must be visually impossible to miss; existing primitives don't deliver that visual weight by default
- Recurring across every kitchen order at v1; structural

**Brand-fidelity-weighted reasons:**
- Visual hierarchy of allergen flags must dominate the slip without making the slip feel "alarming" — restraint-driven brand requires careful red treatment (use `rose` family per `DESIGN.md`, not arbitrary red)
- Patient identity (first name + last initial) needs warmth without disclosing PHI — copy choices matter (e.g., "For: Maria R." vs "Patient: Maria R.")
- Empty space and spacing decisions are load-bearing — the slip must scan in 2 seconds at 24" away

**4-expert panel review required:**
- Pattern-library steward: token discipline (rose family / sizing tokens / `border-radius/md`)
- Information architecture: hierarchy of allergen vs. meal contents
- Accessibility: WCAG AAA contrast on allergen flags; screen-reader behavior (allergen list announces first); large-print mode
- Brand fidelity: warmth of patient identity copy; restraint of allergen treatment; voice
