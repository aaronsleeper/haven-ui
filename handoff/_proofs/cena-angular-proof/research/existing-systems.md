# Existing systems in the "codify-and-emit Angular components" space

Research scout for the Cena pattern-catalog → Angular emission pipeline. The question this report answers: *what already exists, what's adoptable, what's adaptable, what still needs building.* Target: Angular 21.2.13, standalone, zoneless, signal-based, `inject()` DI, control-flow blocks, Firebase Data Connect data layer.

## 1. State of the art

The space breaks cleanly into **five mature families**, each solving an overlapping slice of the problem but none solving the whole.

**Angular Schematics (the native pipe).** Angular ships a first-class template-based code generator. A schematic is a function that mutates a virtual file tree from EJS-like templates; `ng generate` runs them. As of Angular 20.2, scaffolding a new project asks whether to include an AI configuration file ("always use standalone components", "implement lazy loading") and Angular 21 ships signal forms, zoneless migration, and an experimental MCP server in the CLI ([Angular schematics docs](https://angular.dev/tools/cli/schematics-authoring), [InfoQ on Angular 21](https://www.infoq.com/news/2025/11/angular-21-released/)). This is the **native, durable surface** any catalog emission for Angular should ride on or compose with.

**Nx Generators / workspace generators.** Nx extends Angular schematics with an Nx Devkit (`Tree` operations, `formatFiles`, `installPackagesTask`). Custom generators live in the workspace, take a `schema.json` (options + validation) and a `generator.ts` (the tree-mutation function), and are commonly used to "enforce styles or standards at your organization and to generate custom code specific to your organization" ([Nx local generators](https://nx.dev/extending-nx/recipes/local-generators), [angulararchitects.io schematics walkthrough](https://www.angulararchitects.io/en/blog/automating-your-angular-architecture-with-workspace-schematics-part-1-rules-and-parameters/)). Mature shop pattern: enterprise teams (per Çağrı Çakır's [Medium piece](https://medium.com/@cagcak/how-we-standardized-generation-of-feature-modules-by-using-angular-schematics-on-nx-897c24d32608)) standardize feature-module emission this way at scale.

**shadcn-style copy-paste with a registry (Angular instances exist).** shadcn/ui's distribution model is now a public schema: `registry.json` declares items; each registry item is "the fundamental unit of distribution… a JSON object validated by `registryItemSchema` that describes a component, utility, theme, or other code artifact along with its dependencies, files, and configuration requirements" ([registry-item.json docs](https://ui.shadcn.com/docs/registry/registry-item-json)). It's decentralized — namespaced `@namespace/component-name` references, no central registrar ([Introduction to registries](https://ui.shadcn.com/docs/registry)). Two production Angular adaptations now exist: **spartan-ng** (Brain primitives as installed npm + Helm styles copied into your repo, CLI-driven via `@spartan-ng/cli`, requires Angular 17+, Tailwind, supports both Angular CLI and Nx — [spartan.ng CLI docs](https://spartan.ng/documentation/cli), [getting-started](https://dev.to/this-is-angular/getting-started-with-spartanui-shadcn-like-ui-components-for-angular-8df)), and **Volt UI** (Apr 2026, "shadcn philosophy" with a CLI that downloads files, transforms selectors `volt-* → ui-*`, and integrates an MCP server for AI assistants via AnalogJS — [Building Volt UI](https://medium.com/@andriipap/building-volt-ui-bringing-shadcn-ui-philosophy-to-angular-e8a66d152cd5)). **ZardUI** is a third entrant on the same arc ([ZardUI Beta](https://dev.to/samuelrizzondev/zardui-beta-bringing-shadcnuis-philosophy-to-angular-where-you-own-every-line-of-code-2a79)).

**Visual-builder codegen.** Builder.io Visual Copilot generates Angular from Figma; component-library mapping (Figma → existing app components) "is currently available for React, and will soon include Angular" ([builder.io Figma-to-Angular](https://www.builder.io/blog/figma-to-angular)). Plasmic's codegen targets TSX/JSX/React natively; their headless API claims Angular support, but their detailed codegen docs are React-centric ([Plasmic codegen guide](https://docs.plasmic.app/learn/codegen-guide/)). **Both are weak for the Cena shape**: visual-first, not pattern-catalog-first.

**Web-components-as-source.** Lit + Stencil produce framework-agnostic web components and (Stencil) auto-generate Angular/React/Vue wrappers, including standalone-component output as a Stencil output target ([Stencil Angular integration](https://stenciljs.com/docs/angular)). Lightning Design System ships as framework-agnostic CSS blueprints + LWC implementations, explicitly described as "implementation-agnostic" so it supports LWC, Aura, Visualforce, and React simultaneously; SLDS 2 (Spring '25) leans on CSS custom properties ([SLDS](https://www.lightningdesignsystem.com/)). This solves "one source, many frameworks" — but it punts on Cena's actual problem (codifying *application patterns*, not just primitives).

**Adjacent that matters:** Angular Material's **CDK** (the behavior-primitive layer beneath Material — overlays, focus traps, virtual scroll, drag/drop, a11y), explicitly designed as a foundation for custom design systems without prescribing visual styling ([Angular CDK](https://material.angular.dev/cdk/categories)). **Style Dictionary** for tokens — well-understood, multi-platform, can emit component-token JSON, but stops at tokens; it does not emit component code ([Style Dictionary](https://styledictionary.com/info/tokens/)). **Plop / Hygen** as generic JS template scaffolders (EJS or Handlebars + JS hooks) — lighter than Schematics, no AST awareness, "convention over configuration" ([Hygen](https://github.com/jondot/hygen), [Plop](https://plopjs.com/)).

## 2. Who's circling the same problem

Real pieces, named:

- **Angular team, official.** `llms.txt` / `llms-full.txt` published at angular.dev for LLM consumption; Angular 20.2 introduces an experimental **MCP server in the Angular CLI** so AI agents can drive Angular tools first-class; **Web Codegen Scorer** is a new OSS tool that scores AI-generated Angular code against best-practice prompts (signals, standalone, etc.) ([angular.dev/ai/develop-with-ai](https://angular.dev/ai/develop-with-ai), [Angular blog: Beyond the Horizon](https://blog.angular.dev/beyond-the-horizon-how-angular-is-embracing-ai-for-next-gen-apps-7a7ed706e1a3), [The New Stack on Web Codegen Scorer](https://thenewstack.io/new-open-source-tool-from-angular-scores-vibe-code-quality/)). The Angular team is publicly building exactly the prompt-quality infrastructure Cena needs.
- **Alfredo Perez (ng-conf 2025).** "Generating Your First Rules with Cursor for Your Angular Project" + "Creating Angular Guardrails: A Guide to Custom Cursor Rules for Better Code" — describes auto-apply / agent-select / manual cursor rules organized by framework patterns, state management, TestBed, and styling-with-design-systems ([ngconf on Medium](https://medium.com/ngconf/generating-your-first-rules-with-cursor-for-your-angular-project-20a6d013ac9d), [Creating Angular Guardrails](https://medium.com/ngconf/creating-angular-guardrails-a-guide-to-custom-cursor-rules-for-better-code-df5523eafb5d)). This is the closest single artifact to "codify Angular patterns for an AI to follow" published as a community pattern.
- **spartan-ng and Volt UI authors.** Their write-ups are essentially "why we adapted shadcn's CLI+registry to Angular" — they enumerate the same trade-offs Cena will face (CLI vs. copy-paste, primitive vs. styled split, npm-installed vs. project-owned source, schematics integration). Volt UI specifically calls out MCP integration as the AI-assist hook.
- **Andersseen, Apr 2026, "Building Volt UI: Bringing shadcn/ui Philosophy to Angular."** Describes the choice to ship both a CLI (downloads + selector-rewrite + setup) and copy-paste, plus AnalogJS routes that expose the catalog as MCP context ([the article](https://medium.com/@andriipap/building-volt-ui-bringing-shadcn-ui-philosophy-to-angular-e8a66d152cd5)).
- **Sustainable Angular Architectures (Manfred Steyer / angulararchitects.io).** Long-running schematics-for-architecture content — the structural argument for workspace generators as the bones of a maintainable Angular monorepo.

## 3. Solution shapes observed

Four recurring shapes, with sharply different trade-offs.

1. **AST-aware template generators (Schematics / Nx generators).** Tree-based, format-aware, can patch existing files (add an import, update a routing module). Verbose to author. **The right shape for Angular emission** if you need to integrate cleanly with `ng generate` muscle memory and the CLI's existing tooling.
2. **JSON-registry + CLI copy-paste (shadcn / spartan-ng / Volt / Zard).** Each unit is a JSON manifest pointing at source files. CLI resolves dependencies, copies files into the consumer repo, optionally rewrites selectors. **Owns the source after install**, by design. Decentralized — anyone can host a registry. Maps almost directly onto Cena's "copy-and-parameterize the pattern" framing.
3. **Compiler-driven framework wrappers (Stencil, Lit + adapters, SLDS-style blueprints).** One source, many framework outputs. Strong fit when you need to ship *the same UI* to React + Angular + Vue. Weak fit for Cena: Cena isn't multi-framework; it's deeply Angular-21-idiomatic (signals, zoneless, `inject()`). The wrapper overhead works against the catalog's whole point.
4. **LLM-with-rules (cursor rules, `.cursorrules`, `llms-full.txt`, MCP servers).** Pattern is encoded as prose + examples; the LLM does the emission. Angular's official direction. Cheap to author, fragile to drift, hard to make deterministic — output quality is bounded by the model and the prompt's coverage. **Composable with the others, not a replacement.**

**Two patterns recur across all four shapes:**

- **Two layers: behavior primitive vs. styled composition.** shadcn (Radix + theme), spartan (Brain + Helm), Material (CDK + Material), Stencil-built systems (core web component + framework wrapper). Cena's pattern catalog will land here whether designed for it or not.
- **The registry/manifest as contract.** Whether `registry.json`, schematics `schema.json`, or Stencil's output-target config, every mature system encodes "what's in the catalog" as a *machine-readable artifact* separate from the source. That contract is what makes the catalog tooled rather than ad hoc.

**Two patterns that DON'T transfer to a small-team agentic pipeline:**

- **Heavyweight visual builders (Plasmic, Builder).** Their value is letting non-coders design; they pay for that with deep integration ceremony and React-bias. Wrong shape for a 3-human + agent team emitting from a code-first catalog.
- **Universal framework wrappers (Stencil).** The cross-framework cost (web-component lifecycle, ControlValueAccessor bridging for Angular forms, FOUC handling) is paid even when you only target Angular. Cena targets one framework deeply.

## 4. Adoption matrix

| Candidate | Verdict | Notes |
|---|---|---|
| **shadcn `registry.json` / `registry-item.json` schema** | **Adopt verbatim** as the manifest format | Public, decentralized, exactly the shape Cena needs (item = files + dependencies + config); maps cleanly onto a pattern catalog; battle-tested as of CLI v3 (2025-08) with MCP integration |
| **spartan-ng's Brain/Helm split** | **Adopt as architecture** | Their npm-installed primitives + copied-styles split is the codified version of "primitives vs. compositions" Cena's catalog needs; install model can be lifted whole |
| **Angular CDK** | **Adopt verbatim** for behavior primitives | Battle-tested overlay, focus, a11y, virtual-scroll, drag-drop; saves Cena from reinventing primitives and earns standalone-component / signals compatibility for free |
| **Angular Schematics (native CLI)** | **Adopt as one emission surface** | Necessary for any "ng generate from catalog" experience; `ng generate component` already produces standalone + signal-friendly output in 19+ |
| **Style Dictionary** | **Adopt verbatim** for tokens | Tokens-as-source-of-truth; emits the CSS custom properties Angular 21's zoneless + signal patterns already prefer; out of scope for component emission |
| **Angular `llms-full.txt` + Web Codegen Scorer** | **Adopt verbatim** as a prompt-quality floor | Free best-practice grounding for any LLM step in the pipeline; the scorer is an outside-voice quality check |
| **Cursor rules / `.cursorrules` patterns** | **Adapt** | Alfredo Perez's structure (framework / state / testing / styling) is a usable template; Cena's catalog supplies the content |
| **Nx workspace generators** | **Adapt if monorepo** | Worth adopting Nx Devkit if Cena is already in an Nx monorepo; otherwise vanilla Angular Schematics has equivalent reach |
| **Stencil framework-wrapper generation** | **Don't adopt** | Wrong axis — Cena is Angular-deep, not cross-framework; wrapper overhead pays no dividend |
| **Builder.io / Plasmic / Visual Copilot** | **Don't adopt** | React-biased, Angular component-mapping not yet shipped (Builder), visual-builder ceremony is the wrong premise |
| **Hygen / Plop** | **Don't adopt** | No AST awareness, no Angular integration; Schematics gives you both for the same authoring cost |
| **Build still needed** | The **pattern catalog itself** (the Angular-21-idiomatic patterns: standalone shells, signal `input()/output()`, control-flow blocks, `viewChild()`, `inject()` DI, route guards, lazy loading, provider scoping, zoneless change-detection timing, Jasmine specs, Firebase Data Connect bindings) AND the **preference-ledger contract** that parameterizes catalog items per app | No existing catalog covers Angular 21 + zoneless + signals + Data Connect comprehensively; this is the actual authoring work, and the shadcn-registry + Schematics tooling are scaffolding around it |

## 5. Three biggest learnings to absorb before implementation

1. **The manifest is the catalog.** shadcn's quietly-revolutionary move is making the unit of distribution a JSON object validated by a schema, not a directory layout or a package manifest. Adopt `registry.json` / `registry-item.json` directly. It already encodes files, dependencies, registry-dependencies (other catalog items this depends on), tailwind config, css vars, and demo. Every other mature system reinvents this; Cena doesn't have to.
2. **The Brain/Helm split is the catalog's real architecture.** Behavior primitives (installed dependency, npm-versioned, maintained) and styled compositions (copied into the consumer app, owned and editable) are *different things with different lifecycles*. spartan-ng's split — keep Angular CDK as the npm-installed primitive layer, copy your styled compositions as catalog-emitted source — is the load-bearing structural decision. Cena's "pattern catalog emits app code" only works cleanly if patterns are categorized along this axis from day one.
3. **Angular 21's native AI surface is now first-class — ride it, don't reinvent it.** `llms-full.txt`, the Angular CLI MCP server, Web Codegen Scorer, and Angular 21's signal forms + stable zoneless are all shipped in the Angular team's official direction. The Cena pipeline gains durability by treating these as the substrate (catalog items reference Angular's own best-practice prompts; emission ends with a Web Codegen Scorer pass; the pipeline exposes the catalog as MCP context the way Volt UI did via AnalogJS). Building a pattern catalog *against* those flows would mean re-solving problems Google is currently solving.

## 6. Honest unknowns

- **Firebase Data Connect integration patterns in published Angular catalogs:** I did not find specific guidance on codifying Firebase Data Connect bindings as catalog patterns. The signal-based data-flow patterns are documented, but the Data Connect query/mutation generation shape — particularly how it interacts with signals and zoneless — appears to be ahead of the public guidance. Cena will need to author this part from scratch and likely contribute it back.
- **spartan-ng and Volt UI versus Angular 21.2.13:** Coverage I found is mostly Angular 17–20-era. Whether the current Helm/Brain primitives are clean against Angular 21's zoneless + signal-forms is **not verified** from this scout — needs direct experimentation in a sandbox before betting infrastructure on them.
- **Web Codegen Scorer's catalog story:** It exists and scores Angular code, but whether it can score against a custom catalog (i.e., "did the emitted code follow *our* pattern catalog?") versus only Angular's general best practices is **not clear from public docs** and is a critical question for whether it's an honest outside-voice check or just a generic guardrail.
- **MCP-as-catalog-distribution:** Volt UI's AnalogJS MCP routes hint at "expose the catalog to AI assistants as MCP context," and shadcn CLI v3 added MCP integration in 2025-08. The full pattern for "catalog as MCP server an agent queries during emission" is publicly **emerging, not codified** — Cena will be one of the first to formalize it for Angular.
- **ng-conf 2025 talks specifically on catalog → emission pipelines:** I could not surface a single talk titled this way. The patterns exist as community pieces (Perez, spartan-ng author, Volt author) and as Angular-team infrastructure, but the *integrated story* — "here's how we authored a pattern catalog and emit Angular apps from it" — does not appear to be presented end-to-end at a major conference yet. Cena's pipeline is in the leading edge of a real, named space, not behind it.
