# AGENTS.md — Dietary recall slice

For AI coding agents working with this slice. Read [`../AGENTS.md`](../AGENTS.md) first for bundle-wide conventions, then this file.

## What this slice is

Patient-facing 24-hour dietary recall using the multiple-pass methodology (USDA AMPM-style: quick list → forgotten foods → time/occasion → detail/portion → review → submit). Driven by the agent as an interview, not a form. Chat is the primary interaction surface; the right pane holds the running food list (the canonical record).

This is cap-08 of the UConn pilot. HEI scoring is backend-only — the patient never sees a score or component breakdown.

## Shell — critical difference from assessments slice

**This slice uses `layout-agentic-shell` (3-pane, chat-primary).** The assessments slice uses `layout-app-shell-responsive` (deterministic, single content area). These are different shell families with different markup structures. Do not conflate them.

The agentic shell structure:
```html
<div class="app-shell app-shell--agentic">
  <div class="agentic-shell">
    <nav class="panel-nav">…</nav>
    <div class="panel-splitter" aria-hidden="true"></div>
    <main class="panel-chat">…</main>
    <div class="panel-splitter" aria-hidden="true"></div>
    <aside class="panel-content">…</aside>
  </div>
</div>
```

Both shells share the same `.nav-item` family inside their sidebar nav. The outer wrappers differ.

## Runner — key differences from assessment-runner.js

| Dimension | assessment-runner.js | dietary-recall-runner.js |
|---|---|---|
| Input type | Radio options (discrete choices) | Free-text chat (open-ended interview) |
| State per item | Single answer per item | FoodItem with 3 columns (name, when, amount) that populate across passes |
| Pass structure | Linear item index | 7 named passes; each pass populates a different FoodItem column |
| Right pane | No right pane (responsive shell) | `patient-recall-list` that grows + locks across passes |
| Submit gating | All required items answered | Patient signals "looks right" in the review pass |
| Outcome routing | Rule-table computation (screener) | None — submit always routes to confirm |

## Hard invariants

These are not stylistic preferences. Violating one breaks the spec.

- **No score shown to the patient.** HEI scoring runs backend. The confirm receipt shows: day recalled, item count, submitted timestamp, and reassurance copy. Nothing else.
- **No food cards in chat.** The right-pane recall list is the single source of truth. The agent describes additions and edits verbally; chat never renders a structured food card. This prevents drift between the chat thread and the canonical list.
- **No overlay-bottom-sheet or hs-* Preline primitives.** The Preline bottom-sheet is not in the handoff bundle. Mobile sheet behavior is deferred; do not add new Preline wiring.
- **No judgment copy.** Agent messages must honor the no-judgment contract (see README §No-judgment copy rule). Any dynamically generated copy that asks about amounts must include a size anchor ("about a coffee-cup full?") not a measurement demand. Any correction path must never express surprise.
- **Skipped meals: acknowledge and move on.** No probing follow-up. Skipped meals are clinically relevant (the score handles missing items); the agent records the gap and continues.
- **Edit path is always available during passes 2–review.** Both paths (talk to agent / tap right-pane row) must remain active simultaneously. No "edit mode."

## No-Preline constraint

The `overlay-bottom-sheet` and `hs-*` components in the full haven-ui PL require Preline JS, which is not initialized in the handoff bundle (no `import 'preline'` / no `HSStaticMethods.autoInit()`). Do not add Preline-dependent markup to this slice. For inline food-item editing (Path B):

- Use a `<details>` disclosure with field-row inputs, OR
- Append inline `field-row` + commit button elements to the row in DOM on tap

Both are valid. The `field-row` class is present in `../assets/haven.css`.

## Data shapes

See README §Runner contract for the full TypeScript interfaces. The key shape is:

```typescript
interface FoodItem {
  id: string;       // "item-0", "item-1", …
  name: string;     // patient-stated name; free text, no normalization in v1
  when?: string;    // time + occasion; free text
  amount?: string;  // portion + preparation; free text
  addedInPass: number;
}
```

In v1, items are free-text. v2 ties items to USDA FoodData Central (see flow-dietary-recall.md §Future scope).

## Per-pass agent copy contract

Pass-level agent copy is not generated by the runner in static demo mode (it's hard-coded in the HTML exemplars). On Angular port, the runner dispatches `recall:state-change` and the Angular component is responsible for rendering the agent messages per pass. The pass-level copy contract from the flow doc:

| Pass | Agent opener |
|---|---|
| entry | "Hi [name]. Time for your dietary check-in. …Which day do you want to talk about?" |
| pass1 | "Walk me through what you remember eating or drinking [day] — just the names for now, no detail yet. Take your time." |
| pass2 | Paced forgotten-foods probes: drinks besides water? coffee/tea? anything in your coffee? snacks? anything off someone else's plate? (ONE or TWO at a time, not a list) |
| pass3 | "Now let's go through when you ate each thing. What time did you have your first meal or drink [day]?" |
| pass4 | "One more pass — let's get amounts and how things were prepared. Starting with [first item]: about how much did you have?" (with size anchor) |
| review | "Okay — let me read back what we have. …Anything missing or wrong?" (verbal summary grouped by meal; NOT a list dump) |
| confirm | "Thanks — that's it. Your care team will see this at your next session." |

## "I don't remember" handling

Agent offers an anchor prompt first: "What time did you wake up? Did you have anything before you left the house?" If the patient still can't recall, the agent records what exists and moves on. Partial-day data is still clinically useful; the HEI score handles missing items. Do not gate submission on complete data.

## How to know when a porting task is done

For each artifact you port from this slice, completeness means:

- DOM hierarchy + semantic class names match the static exemplar exactly
- `patient-recall-list` updates per-pass (name → name+when → name+when+amount → is-locked)
- Agent messages are generated per-pass using the copy contract above
- Both edit paths (A: talk to agent; B: direct right-pane manipulation) are wired
- No score or HEI number is rendered anywhere on the patient surface
- No Preline bottom-sheet / overlay primitives are used
- No-judgment copy contract is honored in all dynamically generated messages
- `recall:submit` fires with the full `RecallSession` before showing the confirm state
- The confirm receipt uses `log-confirmation-card` (not `assessment-confirmation`) and shows only: day, item count, timestamp, and reassurance copy
