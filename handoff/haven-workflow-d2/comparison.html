D2-ELK vs PlantUML vs tldraw — three-way regime scout

2026-06-04 update · completing step 4 of the emission-regime-scouting rule (spike each regime) — regime 3 (interactive substrate) added

What's new: the prior version of this page compared two regimes — D2-ELK (text→auto-layout) and PlantUML (text→domain-native) — and called PlantUML the winner. The candidate rule that fell out of that work said: "spike the top candidate in each regime." Regime 3 (interactive surface + manual placement) was named but not spiked. This page adds the third column.

The substrate for regime 3 is tldraw — open-source interactive whiteboard whose document format (.tldr JSON) and SVG renderer are both standalone-usable. Lanes are frames; steps are geo shapes (rectangle / diamond); edges are arrows; annotations are notes. Position is first-class in the document (every shape has explicit x, y — no auto-layout). Generated here by a small Python adapter that consumes the same diagram.md spec the PlantUML and D2 emitters read.

tldraw column = second cut (2026-06-04 polish pass). What changed from cut 1: lane labels now render as separate left-edge text shapes (was: tldraw default frame.name at top-left); 220px canvas-top buffer added so callouts no longer crop; arrow size bumped from "s" to "m" for visibility; diamond enlarged to 240×130 so decision text fits; redundant prefixes ("decision-", "action-") stripped from node labels.

Remaining issues you'll see in cut 2 (none are substrate-shape problems; some are tldraw conventions): (1) small "Frame" labels at top-left of each lane frame — tldraw substitutes the default "Frame" name when `name` is empty or single-space, so suppressing them entirely needs a different container shape (use a geo-rectangle as the lane container instead of a frame; ~10 min). (2) Callout sticky-note bounds (tldraw's `note` shape has a min size around 180–200px regardless of text length) overlap with the lane top edge — they don't crop, but they crowd the AGENT-lane nodes. Fix: use a `geo` rectangle with semi-fill for callouts instead of `note`; ~15 min. (3) Right-edge node text wraps oddly because the canvas viewBox grew with the X_BUF + Y_BUF additions. Fix: pad node margins; ~5 min. None of these change the substrate-fitness call.

Decision criteria, pre-named so the call doesn't drift: tldraw wins if its renders are at least as clean as PlantUML's on both cases and the JSON-as-source feels authorable from the spec (the "fill it in like a Mad Lib" test) and position-as-first-class shows up as a visible lever. PlantUML wins if tldraw doesn't beat its renders. Hybrid wins if both viable — PlantUML for batch-render the catalog, tldraw for iteration-loop nudging during scoping.

Read — prior verdict (D2 vs PlantUML)

  • Sparse case: D2-ELK has the two-edge label collision (red/yellow on top of each other) + the two-edge bottom collision (BHN ack / timeout). PlantUML has neither — decision diamonds carry both branch labels cleanly.
  • Dense case: D2-ELK crams the 3-way branch into a tight HUMAN row with stacked labels. PlantUML's switch/case gives each branch its own labeled outgoing edge with no collision.
  • Modifier colors land in both — the styling work survives the engine swap. The PlantUML <<#hex>> stereotype is the equivalent of D2's style.fill.
  • Honest limit on PlantUML: its swim-lanes are vertical bands (not horizontal). Reading direction is top-to-bottom rather than left-to-right. For the use case this repo's diagrams encode (sequential temporal flow), vertical-band reading is fine — arguably more natural for processes that have many decision branches.
  • D2 verdict superseded — PlantUML wins over D2.

Read — tldraw second cut (2026-06-04 polish pass)

  • What landed well in cut 2: lane labels (AGENT/HUMAN/PARTNER/SYSTEM) on left edge as big bold text; callouts no longer crop above canvas; diamond text fits inside the rhombus; lane sequence + node placement faithful to spec coordinates; emphasis treatment visible (blue attestation box, grey ghost audit box, colored callouts).
  • What didn't land cleanly: the small "Frame" labels at top-left of each lane frame (tldraw's default name substitution), the callout sticky-notes still crowd the AGENT-lane nodes because tldraw's `note` shape has a min size around 180px regardless of text length, and (you'll judge this from your own eyes) arrow visibility at preview scale may still be marginal even at size="m". Honest read: these are tldraw convention-vs-our-spec mismatches more than generator bugs. The fixes exist (use geo-rectangle as lane container; use geo for callouts; bump arrow size further) but each adds ceremony and the substrate keeps imposing its own primitives' defaults.
  • The "fill it in like a Mad Lib" test (still holds): the generator is now ~220 LOC. Reads the same `diagram.md` YAML. Maps lane → frame, node → geo, edge → arrow, callout → note. The mapping is straightforward; the polish work is in tuning the primitives' sizes and positions to match the spec's visual conventions, not in the mapping itself.
  • Position-as-first-class lever (still holds): every shape's `x, y` is explicit in the JSON. Nudging a node is a one-field diff. For iteration-loop work this is real.
  • The honest assessment after the polish pass: the renders still aren't at PlantUML's polish level. The gaps that remain are substrate-convention gaps — tldraw's primitives ship with defaults (Frame name substitution, note min-size, frame.name top-left rendering) that don't naturally match our swim-lane convention. Each is fixable but adds layers (custom shape types instead of primitives, or geo-as-container instead of frame-as-container). Cumulative cost is more than 30 more minutes — closer to 2–3 hours to get visual parity with PlantUML.
  • My read on the criteria call after cut 2: the substrate is technically viable (renders the shape, supports explicit position, mapping is clean). But PlantUML's syntax fits our swim-lane case more naturally — its primitives (`|partition|`, `:action:`, `if/elseif/else`) directly name the things we care about, whereas tldraw's primitives (`frame`, `geo`, `arrow`, `note`) name *interactive whiteboard* things we're SPECIALIZING into swim-lane things. The lost thread's PlantUML verdict was right for this specific load-bearing primitive. The candidate rule's step 4 has been honored; the answer is: regime 2 still wins for workflow swim-lanes.
  • Where tldraw still has a place — hybrid (option c): for the *iteration-loop* cases where Aaron/Vanessa want to nudge a diagram during scoping ("move that step left", "this lane goes above that one"), tldraw's explicit-position model is the right tool. For *batch-render the catalog* (the 6 use cases shipped as partner-readable diagrams), PlantUML wins. The two adapters are not mutually exclusive — same `spec_from_markdown.py` parser, two emitters consuming its output.
  • Recommended call: (b) ship `spec_to_puml.py` as the primary adapter. Defer tldraw to a future "interactive-iteration mode" if that mode earns its keep. Reserves (c) — hybrid — as an option for later, not a commitment now.
