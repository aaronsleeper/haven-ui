## Downstream-slot silently softens upstream constraint

- **Date / incident:** 2026-06-10 — cena-apps patient-app V0 shipped with food-image-via-placeholder on meal cards, violating brief §5a's explicit no-food-images constraint.
- **Provenance:** `.claude/rules/pipeline-coverage-gate.md` lines 9–17 (Source incident section, lines 93–95); `~/.claude/plans/ui-pipeline-slot-19-consumes-brief-constraints.md` lines 34–37 (Context — source incident); cena-apps commit `f0f723a` referenced.
- **What the pipeline produced vs. what it should have:** Brief §5a excluded food images. Slot-11 menu wireframe rationalized "no food images" into a `meal-option-card-placeholder` element + "Meal photo not shown" a11y string. Slot-17 inherited the placeholder primitive as the contract. Slot-22 build composed the placeholder. The negative-space constraint never materialized as an acceptance row. The brief said *no images*; the build emitted a styled image-shaped element.
- **Layer the failure landed at:** wireframe (the rationalization moment) → component (inheriting the softened contract) → build (composing it) → acceptance (deriving criteria from wireframe, not brief).
- **Adjacent gate that would have caught it (proposed):** A wireframe → component-spec gate that requires every brief-named constraint to be either positively exercised, **explicitly excluded by inverse** (literally: "zero `<img>` AND zero placeholder-treatment descendants"), or explicitly deferred with brief-owner approval. Pipeline-coverage-gate now ships this discipline; this is the existence proof.

## Brief framing itself is wrong (the upstream gate the new rule didn't catch)

- **Date / incident:** 2026-06-10 → 2026-06-11 — patient-app V0 emitted as a 430px-locked mobile shell centered in empty desktop chrome. Rewind A discovered the failure was *not* a constraint-softening rationalization downstream — the brief's own framing ("mobile-first single-column with graceful upgrade") was the wrong frame.
- **Provenance:** `~/.claude/plans/cena-fable-window-2026-06-11.md` lines 115–117 (Completion Notes 2026-06-11 Rewind A); same plan line 116 (the Gate-0 candidate Aaron flagged as needing to land in `pipeline-coverage-gate.md`); `.claude/rules/pipeline-coverage-gate.md` line 12 (the 430px-locked failure as source incident).
- **What the pipeline produced vs. what it should have:** Slot-30 verifier PASSED 78/78 (no width violations, no h-scroll, no clipped content). The build was a *clean V0 of the wrong brief framing*. Slot-07 IA named `layout-two-pane` as the canonical desktop graceful upgrade but slot-11 deferred it as "not exercised by V0 demo's mobile-first build." All downstream slots were faithful to a framing that asserted a primary/secondary relationship between viewports the product didn't actually want. The product needed to look and function well at every viewport accessed — absolute, not comparative.
- **Layer the failure landed at:** brief (§3a criterion 1 + §6a Browser targets — the philosophical posture was wrong); IA (slot-07 carried "mobile-first" through); wireframe (deferred desktop upgrade); component → build → verification (all faithful to wrong upstream).
- **Adjacent gate that would have caught it (proposed):** A brief-philosophy alignment gate (candidate Gate-0 in `pipeline-coverage-gate.md`): before downstream slots compose, the gate asserts the brief's design philosophy is named explicitly enough to be falsifiable, and any framing language asserting primary/secondary viewport relationships gets flagged for explicit Aaron confirmation. The existing gate enforces fidelity TO the brief; this gate would check whether the brief's framing is the right framing.

## Canon-misalignment — primitive invented or custom-coded instead of using the canonical Haven shape

- **Date / incident:** 2026-06-11 — Rewind C discovered slot-22 build used a `.mobile-app` body class that is not a Haven canon pattern, and custom-coded top-tabs nav instead of using Haven's responsive header + responsive left-panel nav pattern.
- **Provenance:** `~/.claude/plans/cena-fable-window-2026-06-11.md` line 119 (Completion Notes 2026-06-11 Rewind C, structural failures 1 + 2 of 5).
- **What the pipeline produced vs. what it should have:** Slot-22 build composed against primitive names the wireframe used, without verification that those names exist in `COMPONENT-INDEX.md`. `.mobile-app` was novel; the responsive-header + responsive-left-panel-nav pattern already existed in Haven canon and was the right primitive for the surface shape. The pipeline accepted novel-named shapes as if they were canon primitives because no gate verified the wireframe-named primitives against the COMPONENT-INDEX.
- **Layer the failure landed at:** wireframe (named non-canon primitives) → component-spec (treated them as if they were valid) → build (emitted them). The "pattern-library-first / copy-don't-generate" rule in `Lab/haven-ui/CLAUDE.md` exists to prevent this but wasn't gated in the cena-apps pipeline.
- **Adjacent gate that would have caught it (proposed):** A canon-alignment gate at slot-11 → slot-17 transition that runs the haven-mapper wireframe-vs-PL delta review programmatically: every wireframe-named primitive must resolve to (a) an existing PL fragment, (b) a decomposition into existing primitives, or (c) an explicit promotion-to-PL request that lands a fragment + COMPONENT-INDEX row + components.css class BEFORE the slot proceeds. Sibling to the haven-primitive-codification rule.

## UX-adequacy gap — persona-calibration missing for vulnerable user populations

- **Date / incident:** 2026-06-11 — Rewind C found the patient-app V0 build had light UX copy and weak affordances for the non-tech-savvy / mixed-tech-literacy / partly-ESL patient population the UConn pilot targets.
- **Provenance:** `~/.claude/plans/cena-fable-window-2026-06-11.md` line 119 (Completion Notes Rewind C, structural failure 5 of 5); `~/.claude/plans/cena-uconn-patient-app-canonical-run.md` line 65 (population frame: HIV+/food-insecure adults; mixed/low tech-literacy; some ESL; sensitive — dignity, no judgment/surveillance/gamification — non-negotiable).
- **What the pipeline produced vs. what it should have:** The build was syntactically clean and rendered correctly but the copy, affordance density, and reading-level were calibrated for a generic-user audience rather than the specific vulnerable population the surface serves. There was no slot in the pipeline that asked: "does this read for someone with low tech literacy, in a state of food insecurity, possibly ESL, possibly in distress?" Persona was *named* in slot-01 brief but not *applied* as a gate at any downstream slot.
- **Layer the failure landed at:** all post-IA slots (wireframe / component / build) had no persona-calibration check; verification slot-30 measured structure (widths, contrast, no-clipping) not adequacy-to-population.
- **Adjacent gate that would have caught it (proposed):** A persona-calibration discipline gate — either a dedicated slot or a checklist applied at slot-17 component-spec authoring and again at slot-30 verification. Source-of-design at design-celebrated companies with vulnerable users (Airbnb trust-and-safety, Headspace acute-state users) is the Phase 1-R.b research target. The discipline encodes: copy reading-level check, affordance density check, accessibility floor explicit, language register named.

## Component-level layout discipline gap — wireframe names primitive but not its internal layout

- **Date / incident:** 2026-06-11 — Rewind C found `patient-focus-card` rendered with horizontal layout that broke on mobile (text wrapping into unreadable shapes). The wireframe named the primitive but didn't specify its internal layout against the viewport range.
- **Provenance:** `~/.claude/plans/cena-fable-window-2026-06-11.md` line 119 (Completion Notes Rewind C, structural failure 3 of 5: "patient-focus-card horizontal layout breaks on mobile (text wrapping; UX wasn't tasked with thinking through that component)").
- **What the pipeline produced vs. what it should have:** Wireframe authored to slot-11 syntax named the primitive (`patient-focus-card`) but the wireframe-syntax encodes primitive *names*, not internal *layout*. Component-spec inherited the name. Build emitted the primitive. No slot in between asked "how does this primitive's internal layout adapt across the viewport range named in the brief?" The wireframe-named primitive's *contract* was treated as the design output; its *behavior across viewports* was not.
- **Layer the failure landed at:** wireframe (under-specified internal layout); component-spec (didn't probe the gap); build (emitted what was named).
- **Adjacent gate that would have caught it (proposed):** A component-layout gate at slot-17: for every wireframe-named primitive, the component spec must specify (a) internal layout at each named viewport breakpoint, OR (b) explicit reference to the primitive's PL fragment which already specifies this. The PL fragment is the authority; the spec must dereference it, not just name-drop.

## Consumer-surface invisibility — primitive landed in components.css + COMPONENT-INDEX but consumer pipeline never wired

- **Date / incident:** 2026-06-07 — directive-canon-alignment slice landed `.escalation`, `.glossary-term`, `.glossary-def`, `.review-marker` in components.css + COMPONENT-INDEX + brand spec via slice 3. The SoP HTML render pipeline (`sop-html-emit.sh`) was authored without reference to these primitives or to the existing `pattern-library/pages/doc-care-coordinator-sop.html` spec, producing 52rem prose instead of haven document district output.
- **Provenance:** `.claude/rules/haven-primitive-codification.md` lines 5, 14, 101–103 (Source section); `~/.claude/plans/cena-fable-window-2026-06-11.md` line 127 (Context — what failed 2026-06-07); retro `retro-2026-06-07.md` lines 121–128 (Pattern: Haven primitive visibility to consumer surfaces).
- **What the pipeline produced vs. what it should have:** Primitives existed in spec (PL fragment + index row + brand spec) but the consumer surface (SoP render pipeline) was authored without reference to them. SoP markdown bearing `:::escalation` and `:::glossary-term` directives rendered as plain prose because the pipeline's handler had no branches for those names. The primitive lived in the spec but was structurally invisible to the surface meant to consume it.
- **Layer the failure landed at:** consumer-surface wiring (render pipeline) — a layer *outside* the spec-side chain that the existing slice ceremony owned. The five-place check (PL + index + brand spec + consumer-surface references + render-pipeline verification) is what closes this gap.
- **Adjacent gate that would have caught it (proposed):** The five-place codification gate that `haven-primitive-codification.md` now encodes: every new primitive must reach codification completeness across PL fragment, COMPONENT-INDEX row, brand spec entry, consumer-surface references (wireframe templates, output-type contracts, agent-discoverable indexes, render pipelines), and render-pipeline verification — before being marked done.

## Canon-propagation drift — registry of consumers absent; canon changes hit a subset silently

- **Date / incident:** 2026-05-30 — Sand-warming color canon update successfully pushed to 7 known consumers; missed 14 unregistered consumers. Discovered via post-hoc audit, not by any system surfacing the gap at change time.
- **Provenance:** `~/.claude/plans/haven-canon-propagation-workflow.md` lines 16, 38–42 (Success criteria + Done section); parent plan `~/.claude/plans/cena-sand-warmth-coordinated-push.md`.
- **What the pipeline produced vs. what it should have:** Canon change applied as a 7-consumer push based on memory ("who consumes this?"). The actual consumer graph had 21 entry points. The 14 missed consumers continued running stale canon (old hex values, old token names) silently — drift invisible until the next surface comparison surfaced it. No registry, no scan, no falsifiable check.
- **Layer the failure landed at:** canon-distribution layer — adjacent to the per-surface emission pipeline. This is a multi-consumer / multi-pipeline coordination gap, not a per-pipeline gap.
- **Adjacent gate that would have caught it (proposed):** A canon-consumer registry + `.haven-consumer.json` per consumer + deterministic `sync.mjs` orchestrator + falsifiable `/qa` check for unregistered-consumer scan. The plan ships this discipline; it is the coordination-layer sibling to per-pipeline gates. Maps to the Phase 1-R.b coordination-spine question (Block E across the 5 companies).

## Cross-cutting observations

A reader of the synthesis pass (Workflow Designer, Phase 1-R.c) will notice that **most** of the named failure shapes above are instances of one of three meta-shapes:

1. **Constraint or canon exists upstream but downstream slot does not consume it.** Food-image-via-placeholder, primitive-invisible-to-render-pipeline, acceptance-only-from-wireframes, canon-propagation-misses-consumers, invented-mock-data, canon-misalignment-uses-non-PL-primitives — all variants. The cure shape is the same: **dereference the upstream artifact mechanically; fail loudly when it is absent or stale**. This is the design pattern `generative-determinism.md` names; the failures are instances where the contract wasn't enforced.

2. **Brief or framing is wrong before downstream fidelity even matters.** Brief-philosophy-mobile-first, mode-confusion-Mode-B-vs-Mode-A — both fail upstream of the chain everyone is auditing. The cure shape is a Gate-0 / pre-pipeline check: name the brief's framing explicitly enough to be falsifiable; name the dispatch's mode explicitly enough to route correctly.

3. **Pipeline existed; pipeline wasn't run.** Pipeline-fork-collapse, expert-skips-slots, catalog-first-miss-in-synthesis-class-work — three different ways the agent routed around the canonical chain entirely. The cure shape is routing-side: thread-entry gate that names "what pipeline am I in" before doing any artifact work, and a forced catalog query when the verb implies an existing artifact may already exist.

The Q5b mitigation ledger should cluster on these three meta-shapes and ask the v2 architecture to either prevent each cluster or name explicitly which it accepts and why.

## Honest limits of this mining

- **Self-report is weak.** Most evidence comes from agent-authored Completion Notes, retros, and Aaron-annotated plans. The agent (me, prior sessions) could have under-counted failures it missed entirely, or under-described failures it preferred not to surface. Spot-checking against Aaron's `@Aaron` annotations + the parent plan's Rewind C diagnosis (which Aaron framed personally) gives some confidence but not full coverage.
- **Recency bias.** The richest source is the 2026-06-11 parent plan, which describes failures from 2026-06-07 → 2026-06-11. Earlier failures in haven-ui's history (March-April 2026, pre-canonical-pipeline-collapse) are under-represented because their Completion Notes are thinner. If a failure shape recurred across a longer span, this log probably names only the most recent instance.
- **Plan-titled-failures vs. plan-titled-features bias.** Plans titled "X build" with completion notes naming what failed are over-represented; plans titled "X primitive" or "X feature" where failures happened mid-build but weren't reflected in Completion Notes are under-represented.
- **No retro-log forensics for the cena-fable-window thread itself.** The richest single source describes failures that happened in the same session-arc this miner runs in. Future retros may surface shapes from this very week that this miner did not catch because they post-dated its read of the source.
